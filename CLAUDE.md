# CLAUDE.md — ProjectVault

This is a personal hobby project brainstorm and planning space. Claude's job here is to help explore, design, and document project ideas — not to write production code.

---

## Folder Structure

```
ProjectVault/
├── CLAUDE.md                      ← this file
├── README.md                       ← master index of all ideas + workflow guide
├── _shared/
│   ├── Stack-Preferences.md      ← baseline tech preferences and standing decisions
│   └── Workflow-Prompts-Log.md   ← log of prompts that caused structural workflow changes
└── ideas/
    └── [idea-slug]/
        ├── Overview.md            ← elevator pitch, goals, non-goals, status
        ├── Stack.md               ← tech choices with pros/cons tables
        ├── Architecture.md        ← system design, data model, component breakdown
        ├── Phases.md              ← implementation phases and feature backlog
        ├── Design-Spec.md         ← consolidated implementation spec (source of truth)
        ├── ProjectInitialize.md   ← generated by /vault-init [slug] when idea moves to a real repo
        └── Research/
            ├── *.html             ← interactive design artifacts from brainstorming sessions
            └── *.md               ← spike notes, references, findings
```

---

## Per-Idea Folder Rules

- Slug format: `kebab-case`, short and descriptive (e.g., `habit-tracker`, `cli-budget-tool`)
- Every idea must have at least `Overview.md` before other files are created
- `Stack.md` always includes comparison tables — not just a final answer
- `Architecture.md` includes a Mermaid diagram when the system has more than one component
- `Design-Spec.md` is the consolidated spec written after all design decisions are made — it is the source of truth for implementation
- `Phases.md` is optional for early-stage ideas; add it once scope is reasonably defined
- `ProjectInitialize.md` is generated by `/vault-init [slug]` when an idea is ready to move to a real repo — do not create manually

## Brainstorming Workflow

When asked to start or explore a new idea, follow this sequence:

### 1. Clarify the idea (before creating any files)
- Invoke the `superpowers:brainstorming` skill
- Ask clarifying questions one at a time: audience, scope, core interaction model, constraints
- Offer the visual companion for questions that benefit from mockups or diagrams
- Do not create files until the idea is at least loosely understood

### 2. Propose approaches
- Present 2–3 architectural approaches with trade-offs
- Make a clear recommendation with reasoning
- Get approval before proceeding

### 3. Present design sections
- Walk through the design one section at a time: architecture → data model → frontend → API/infra
- Use the visual companion (browser) for architecture diagrams and component trees
- Get approval after each section before moving on

### 4. Create idea files (in this order)
1. `ideas/[slug]/Overview.md` — pitch, goals, non-goals, status
2. `ideas/[slug]/Stack.md` — all tech decisions with comparison tables
3. `ideas/[slug]/Architecture.md` — Mermaid diagrams + API tables
4. `ideas/[slug]/Phases.md` — phased build plan
5. `ideas/[slug]/Design-Spec.md` — consolidated spec tying everything together
6. `Research/*.html` — save all interactive brainstorming HTML artifacts as standalone files
7. Update `README.md` — add row to the ideas table

### 5. ProjectInitialize.md (when ready to build)
- Created when the idea moves from Exploring → Building
- Run `/vault-init [slug]` — the `projectvault-init-repo` skill generates this file from the existing design docs

## Design-Spec.md Structure

The spec is a consolidated reference document. It must contain:

1. **Product Overview** — one-paragraph description + core differentiators
2. **Users & Roles** — who can do what, auth approach
3. **Core Model** — the main domain concept (e.g., guide structure, enums, JSONB shape)
4. **Data Model** — all tables with fields, types, constraints
5. **API** — all endpoints with method, path, auth, notes
6. **Frontend** — routes (SSR vs CSR), key components and their responsibilities
7. **Infrastructure** — deployment diagram, IaC modules
8. **Phased Build** — what ships in each phase
9. **Out of Scope** — explicit list of what is NOT being built

## Research/ HTML Artifacts

During brainstorming, interactive HTML mockups are generated via the visual companion. These are saved as standalone static files in `Research/`. Each file:
- Is fully self-contained (no server required) — open directly in a browser
- Responds to dark/light mode
- Retains click interactivity for selecting options
- Is named semantically: `layout-types.html`, `design-architecture.html`, etc.

Standard artifact names to expect after a full brainstorm:
| File | Contents |
|------|----------|
| `layout-types.html` | Core interaction model options |
| `authoring-modes.html` | Creator editor options |
| `learner-experience.html` | Learner perspective per layout |
| `design-architecture.html` | System architecture diagram |
| `design-datamodel.html` | Database schema + JSONB shapes |
| `design-frontend.html` | Component tree + route structure |
| `design-api-deploy.html` | API endpoints + infra diagram |

## How to Update This Space

- When a new idea starts: add a row to `README.md`, set status to `Seed`
- When status changes: update both `Overview.md` frontmatter and `README.md`
- When stack preferences change: update `_shared/Stack-Preferences.md` and note the reason
- When an idea moves to a repo: run `/vault-init [slug]` — generates `ProjectInitialize.md` and sets status to `Building`

## Skill Overrides

These override default Claude behavior and `superpowers:brainstorming` in this vault:

- When a new idea is described → invoke `projectvault-new-idea`. **Do NOT invoke `superpowers:brainstorming`.**
- When multiple ideas are listed in one message or `/intake` is used → invoke `projectvault-rapid-intake`
- `/design [slug]` → invoke `projectvault-new-idea` scoped to that existing Seed-status idea
- `/vault-init [slug]` → invoke `projectvault-init-repo` to generate `ProjectInitialize.md` and mark idea as Building
- `/vault-status` → read `README.md` and render a formatted status table inline
- `/vault-resync` → invoke `vault-resync` to audit and repair harness drift (CLAUDE.md, skills, README, idea folders)

## Prompter Profiles

Stored at `_shared/Prompter-Profile.md`. Read this at the start of every session.

If the file is missing: run the first-time setup flow (defined in `projectvault-new-idea` skill) before doing anything else. Write the profile, then resume the request.

| Profile field | Effect |
|---------------|--------|
| `technical_depth` | Controls how infra/stack questions are framed — or whether they're asked at all |
| `skip_questions` | Topics to never ask about (e.g., infrastructure, IaC, auth) |
| `assumed_stack: true` | Use Stack-Preferences defaults without asking any stack questions |
| `rapid_mode_default: true` | Always use rapid-intake (stubs only), never full design flow |
| `prefer_visuals: true` | Offer visual companion early for any idea with UI or architecture |
| `max_clarifying_questions` | Hard cap on Q&A rounds per idea |
